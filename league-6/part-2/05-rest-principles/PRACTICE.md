## Задача 1 — Ревью плохого API

Ниже — API интернет-магазина, спроектированный стажёром. В нём **10 нарушений** REST-принципов.

Найди каждое и предложи исправление. Формат ответа — таблица.

```
POST   /getProducts                          → 200 OK (список товаров)
GET    /createProduct                         → 200 OK (создаёт товар)
GET    /deleteUser/15                         → 200 OK (удаляет юзера)
PUT    /product/update                        → 200 OK (обновляет товар целиком)
POST   /api/v1/Order                          → 200 OK (создаёт заказ)
GET    /api/v1/product?id=5                   → 200 OK (один товар по id)
PATCH  /api/v1/products/5/changeStatus        → 200 OK (меняет статус товара)
DELETE /api/v1/products/5                     → 200 OK (удаляет товар)
POST   /api/v1/products                       → 200 OK (создаёт товар)
GET    /api/v1/users/5/orders                 → 200 OK (заказы юзера)
```

- Неправильные/неинформативные коды ответов
- Ресурс не в URL, а в query-параметрах без причины
- Отсутствие вложенности или лишняя вложенность

## Задача 2 — Спроектируй API с нуля

Ты проектируешь REST API для **сервиса доставки еды** (типа Яндекс.Еда / Delivery Club).

**Сущности:**

- **Рестораны** — название, адрес, рейтинг, режим работы
- **Меню** — блюда привязаны к ресторану (название, цена, категория, доступность)
- **Заказы** — привязаны к пользователю и ресторану (состав, статус, адрес доставки, время)
- **Курьеры** — имя, статус (свободен/занят), текущий заказ

**Что нужно сделать:**

Спроектируй эндпоинты для следующих действий. Для каждого укажи: **метод**, **URL**, **коды ответов** (успех + основные ошибки).

**Рестораны:**
1. Получить список ресторанов (с фильтрацией по кухне)
2. Получить один ресторан
3. Получить меню ресторана

**Заказы:**
4. Создать заказ
5. Получить заказ по id
6. Получить все заказы текущего пользователя
7. Отменить заказ

**Курьеры (админка):**
8. Получить список свободных курьеров
9. Назначить курьера на заказ
10. Обновить статус курьера

**Формат ответа:**

| # | Действие | Метод | URL | Успех | Ошибки |
|---|----------|-------|-----|-------|--------|
| 1 | Список ресторанов | ... | ... | ... | ... |
| ... | ... | ... | ... | ... | ... |

**Подсказки (открой после попытки):**

<details>
<summary>Фильтрация</summary>
Фильтрация — это всегда query-параметры, не отдельные эндпоинты. Не делай `/restaurants/italian` — делай `/restaurants?cuisine=italian`.
</details>

<details>
<summary>Отмена заказа</summary>
Отмена — это изменение статуса, а не удаление ресурса. Подумай, какой метод и URL подойдут.
</details>

<details>
<summary>Назначение курьера</summary>
«Назначить курьера на заказ» — это обновление заказа или создание нового ресурса-связи? Оба варианта имеют право на жизнь, но один чище.
</details>

## Задача 3 — Спорные кейсы

В REST нет единственно правильного ответа на всё. Ниже — реальные дилеммы, с которыми сталкиваются разработчики. Для каждой: выбери вариант и **обоснуй** — почему именно он, и какие у него минусы.

**Кейс 1 — Аутентификация:**

Пользователь логинится. Как оформить?

- **Вариант A:** `POST /login`
- **Вариант B:** `POST /sessions`
- **Вариант C:** `POST /auth/token`

**Кейс 2 — Лайк:**

Пользователь лайкает пост. Как оформить?

- **Вариант A:** `PUT /posts/5/like`
- **Вариант B:** `POST /posts/5/likes`
- **Вариант C:** `PATCH /posts/5` с телом `{"liked": true}`

**Кейс 3 — Поиск:**

Поиск товаров с кучей параметров (цена, категория, рейтинг, сортировка, пагинация).

- **Вариант A:** `GET /products?category=phones&min_price=100&sort=rating`
- **Вариант B:** `POST /products/search` с телом `{"category": "phones", "min_price": 100, ...}`

**Кейс 4 — Массовые операции:**

Админ хочет заблокировать 50 пользователей разом.

- **Вариант A:** 50 отдельных запросов `PATCH /users/{id}` с телом `{"blocked": true}`
- **Вариант B:** `PATCH /users` с телом `{"ids": [1, 2, ...], "blocked": true}`
- **Вариант C:** `POST /users/bulk-block` с телом `{"ids": [1, 2, ...]}`

**Кейс 5 — Вложенность:**

Получить отзывы конкретного товара конкретного магазина.

- **Вариант A:** `GET /shops/3/products/7/reviews`
- **Вариант B:** `GET /reviews?shop_id=3&product_id=7`
- **Вариант C:** `GET /products/7/reviews`

**Формат ответа:**

Для каждого кейса:
```
Выбор: Вариант X
Почему: ...
Минусы: ...
```
