## Задача 1 — Ревью плохого API

| # | Было | Проблема | Стало |
|---|------|----------|-------|
| 1 | `POST /getProducts` | Получение данных = GET, глагол `get` в URL не нужен | `GET /api/v1/products` → 200 |
| 2 | `GET /createProduct` | Создание = POST, глагол `create` в URL не нужен | `POST /api/v1/products` → 201 |
| 3 | `GET /deleteUser/15` | Удаление = DELETE, глагол `delete` в URL не нужен, GET не должен менять данные | `DELETE /api/v1/users/15` → 204 |
| 4 | `PUT /product/update` | Глагол `update` не нужен (PUT сам означает обновление), нет id ресурса, `product` должен быть во множественном числе | `PUT /api/v1/products/5` → 200 |
| 5 | `POST /api/v1/Order` | Регистр — `Order` с большой буквы, ресурсы в URL всегда lowercase и множественное число | `POST /api/v1/orders` → 201 |
| 6 | `GET /api/v1/product?id=5` | Конкретный ресурс по id — часть пути, не query-параметр. `product` → `products` | `GET /api/v1/products/5` → 200 |
| 7 | `PATCH .../changeStatus` | Глагол `changeStatus` в URL — PATCH уже означает частичное обновление | `PATCH /api/v1/products/5` с телом `{"status": "..."}` → 200 |
| 8 | `DELETE .../products/5` → 200 | Код 200 при удалении — формально допустимо, но 204 No Content точнее: тело ответа не нужно | `DELETE /api/v1/products/5` → 204 |
| 9 | `POST .../products` → 200 | Код 200 при создании — должен быть 201 Created | `POST /api/v1/products` → 201 + заголовок `Location` |
| 10 | `GET .../users/5/orders` → 200 | Этот эндпоинт **корректный** — вложенность оправдана, метод и URL правильные. Не всё сломано 😉 | Оставить как есть |

### Что здесь важно

- **Глаголы в URL** — главный признак плохого REST. Метод (GET/POST/PUT/DELETE) уже несёт смысл действия, URL описывает только ресурс.
- **Коды ответов** — не всё 200. Минимальный набор: 200 (ОК), 201 (создано), 204 (удалено, тело пустое), 400 (плохой запрос), 404 (не найдено).
- **Множественное число** — `products`, не `product`. Коллекция ресурсов — это множество.
- **Консистентность** — если один ресурс `products`, другой не может быть `Order`.

## Задача 2 — Спроектируй API с нуля

| # | Действие | Метод | URL | Успех | Ошибки |
|---|----------|-------|-----|-------|--------|
| 1 | Список ресторанов | GET | `/api/v1/restaurants?cuisine=japanese` | 200 | 400 (невалидный параметр) |
| 2 | Один ресторан | GET | `/api/v1/restaurants/{id}` | 200 | 404 |
| 3 | Меню ресторана | GET | `/api/v1/restaurants/{id}/menu` | 200 | 404 (ресторан не найден) |
| 4 | Создать заказ | POST | `/api/v1/orders` | 201 | 400 (невалидные данные), 422 (ресторан закрыт, блюдо недоступно) |
| 5 | Получить заказ | GET | `/api/v1/orders/{id}` | 200 | 404, 403 (чужой заказ) |
| 6 | Заказы пользователя | GET | `/api/v1/users/me/orders` | 200 | 401 (не авторизован) |
| 7 | Отменить заказ | PATCH | `/api/v1/orders/{id}` с телом `{"status": "cancelled"}` | 200 | 404, 409 (заказ уже доставлен) |
| 8 | Свободные курьеры | GET | `/api/v1/couriers?status=available` | 200 | 403 (не админ) |
| 9 | Назначить курьера | PATCH | `/api/v1/orders/{id}` с телом `{"courier_id": 5}` | 200 | 404, 409 (курьер занят) |
| 10 | Статус курьера | PATCH | `/api/v1/couriers/{id}` с телом `{"status": "busy"}` | 200 | 404 |

### Разбор спорных решений

**Пункт 3 — Меню как вложенный ресурс.** `/restaurants/{id}/menu` — меню принадлежит ресторану, вложенность оправдана. Альтернатива `/menu?restaurant_id=5` тоже работает, но менее выразительна.

**Пункт 6 — `/users/me/orders`.** `me` — распространённый паттерн, означающий «текущий авторизованный пользователь». Не нужно передавать свой id — сервер определяет по токену.

**Пункт 7 — Отмена как PATCH, а не DELETE.** Отмена заказа ≠ удаление. Заказ остаётся в системе (история, аналитика, возвраты). Это смена статуса → PATCH. `409 Conflict` — когда действие невозможно из-за текущего состояния ресурса (заказ уже доставлен).

**Пункт 9 — Назначение как обновление заказа.** Курьер — это атрибут заказа. Альтернатива `POST /api/v1/assignments` (отдельный ресурс-связь) — тоже валидна, но сложнее без необходимости.

## Задача 3 — Спорные кейсы

### Кейс 1 — Аутентификация

**Рекомендация: Вариант B — `POST /sessions`**

Логин — это создание сессии. Сессия — ресурс. `POST /sessions` — чистый REST: создаём ресурс, получаем токен. Логаут — `DELETE /sessions/current`.

`POST /login` — проще и понятнее для фронтенда, но это глагол в URL. Большинство реальных API так и делают — прагматизм побеждает пуризм.

`POST /auth/token` — ближе к OAuth2 и хорош, если у тебя несколько способов аутентификации.

**Минусы B:** менее очевиден для новичков. Команде придётся объяснять, что «создать сессию = залогиниться».

---

### Кейс 2 — Лайк

**Рекомендация: Вариант B — `POST /posts/5/likes`**

Лайк — это отдельный ресурс. Создать лайк = `POST /posts/5/likes`. Убрать лайк = `DELETE /posts/5/likes` (или `/posts/5/likes/mine`). Получить все лайки = `GET /posts/5/likes`.

Вариант A (`PUT /posts/5/like`) — глагол в URL.

Вариант C (`PATCH /posts/5` с `{"liked": true}`) — лайк не является свойством поста, это действие пользователя. PATCH поста — для редактирования текста/заголовка.

**Минусы B:** оверинжиниринг для простого флага. Если лайк — это просто bool без дополнительных данных, отдельный ресурс может быть избыточен.

---

### Кейс 3 — Поиск

**Рекомендация: Вариант A — `GET /products?...`** для стандартных случаев.

GET с query-параметрами — идиоматичный REST. Кешируется, можно поделиться ссылкой, идемпотентный.

**Когда переходить на B:** если параметров настолько много, что URL превышает ~2000 символов (лимит браузеров), или если фильтры содержат сложные вложенные структуры (гео-полигоны, массивы объектов). Elasticsearch API, например, использует POST для поиска.

**Минусы A:** URL может стать нечитаемым при 10+ параметрах. Не подходит для сложных фильтров.

**Минусы B:** POST не кешируется, не идемпотентный, не вставишь в закладки.

---

### Кейс 4 — Массовые операции

**Рекомендация: Вариант C — `POST /users/bulk-block`**

50 отдельных запросов (A) — убьют производительность. Одна транзакция надёжнее, чем 50 независимых.

`PATCH /users` (B) — опасно. Этот эндпоинт выглядит как «обновить всех пользователей». Легко ошибиться, забыв передать `ids`.

`POST /users/bulk-block` (C) — да, глагол в URL. Но массовые операции — исключение, где чистый REST не работает. Google, Stripe, GitHub — все используют специальные эндпоинты для batch-операций. Прагматизм > пуризм.

**Минусы C:** нарушает принцип «без глаголов в URL». Для каждой новой массовой операции — отдельный эндпоинт.

---

### Кейс 5 — Вложенность

**Рекомендация: Вариант C — `GET /products/7/reviews`**

Отзывы принадлежат товару. Один уровень вложенности — читаемо и логично. Магазин для фильтрации отзывов не нужен — товар уже принадлежит магазину.

Вариант A (`/shops/3/products/7/reviews`) — три уровня вложенности. Если знаешь `product_id`, зачем дублировать `shop_id`? URL становится хрупким: поменялся магазин у товара — сломались все ссылки.

Вариант B (`/reviews?...`) — работает, но теряется иерархия. Отзывы существуют только в контексте товара.

**Практическое правило:** максимум 2 уровня вложенности. Глубже — используй query-параметры.

**Минусы C:** если нужны отзывы по магазину (все отзывы всех товаров магазина) — придётся делать отдельный эндпоинт или фильтр.
