# Solution — Postman практика

---

## Как импортировать коллекцию из JSON

В папке задания лежит файл `Practice.postman_collection.json` — это готовая коллекция со всеми запросами.

**Импорт в Postman:**

1. Открой Postman.
2. Нажми **Import** (кнопка в левом верхнем углу) или меню **File → Import**.
3. Выбери один из вариантов:
   - **Upload Files** — нажми и выбери файл `Practice.postman_collection.json` на диске;
   - либо перетащи файл в окно Postman.
4. В списке появится коллекция **Practice** — нажми **Import** (если спрашивает подтверждение).
5. В боковой панели слева коллекция **Practice** появится со всеми 6 запросами. Можно сразу отправлять запросы через **Send**.

После импорта все запросы готовы к использованию: URL, метод, тело и заголовки уже заданы.

## Часть 1 — Первый запрос

**Статус-код:** `200 OK` — запрос успешен

**Формат ответа:** JSON — массив объектов

**Сколько объектов:** 100 постов

---

## Часть 2 — Query-параметры

**Что изменилось:** вместо 100 постов вернулся один — с `id: 3`

Query-параметры добавляются к URL после `?`:
```
https://jsonplaceholder.typicode.com/posts?id=3
```

Postman во вкладке Params сам собирает эту строку — удобнее чем писать вручную, особенно когда параметров несколько.

**При id=99:** вернулся пустой массив `[]` — пост с таким id не существует, но сервер ответил `200`. Это нормально — ошибки нет, просто результат пустой.

---

## Часть 3 — Заголовки

**Заголовки которые Postman добавляет автоматически:**
- `User-Agent: PostmanRuntime/...` — говорит серверу что запрос от Postman
- `Accept: */*` — клиент принимает любой формат ответа
- `Accept-Encoding: gzip, deflate, br` — поддерживаемые форматы сжатия
- `Connection: keep-alive` — держать соединение открытым

**Для чего нужен Content-Type:**

Говорит серверу в каком формате передаются данные в теле запроса. Например `Content-Type: application/json` означает — тело содержит JSON. Без этого заголовка сервер не знает как парсить тело.

---

## Часть 4 — POST-запрос с телом

**Статус-код:** `201 Created` — ресурс успешно создан

**Что вернулось в ответе:**
```json
{
  "title": "Мой первый пост",
  "body": "Текст поста",
  "userId": 1,
  "id": 101
}
```

Сервер вернул тот же объект но с добавленным полем `id: 101` — это id только что созданного ресурса. В реальном API именно этот id потом используется для обращения к записи.

Данные передаются через POST а не GET потому что в GET нет тела — параметры пришлось бы передавать в URL, что небезопасно и неудобно для больших данных.

---

## Часть 5 — PUT и DELETE

**PUT — ответ сервера:**

Сервер вернул объект с обновлёнными полями и статус `200 OK`. PUT полностью заменяет ресурс переданными данными.

**DELETE — статус-код:** `200 OK`

jsonplaceholder возвращает `200` с пустым телом `{}`. В реальных API чаще возвращают `204 No Content` — запрос выполнен, возвращать нечего.

---

## Часть 6 — Авторизация

Postman автоматически добавил заголовок:
```
Authorization: Bearer my-secret-token
```

Это стандартный способ передачи токена — в заголовке `Authorization` со словом `Bearer` перед токеном. Сервер на своей стороне читает этот заголовок и проверяет токен перед обработкой запроса.

В данном случае jsonplaceholder токен не проверяет — но в реальных API без правильного токена вернётся `401 Unauthorized`.
